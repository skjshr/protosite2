# 画面・コンポーネント構成 詳細設計

## 1. 目的

- 画面とコンポーネントの責務を明確にする
- ロジックを UI に埋め込まないための境界を定義する
- Codex が画面を実装する際の指針とする

## 2. 画面一覧

| パス | 画面名 | 認証要否 | オンボーディング要否 |
|------|--------|---------|------------------|
| `/login` | ログイン | 不要 | 不要 |
| `/onboarding` | ユーザー名設定 | 必要 | 不要 |
| `/` | ホーム（セッション + コレクション） | 必要 | 必要 |

ミドルウェア (`src/middleware.ts`) が認証・オンボーディングのリダイレクトを制御する。

コレクション（図鑑）は別ページではなく、ホームのタブとして実装する。

## 3. 各画面の責務

### 3-1. ログイン画面 `/login`

**責務：** プロバイダ選択の表示とサインインの起動

- Google / GitHub / X (Twitter) / 開発用ログインのボタンを表示する
- `signIn(provider)` を呼ぶだけ。認証ロジックは `src/lib/auth-options.ts` に委ねる
- ログイン済みの場合は `/` にリダイレクトする

### 3-2. オンボーディング画面 `/onboarding`

**責務：** ユーザー名の初回設定

- テキスト入力 + 送信ボタン
- `POST /api/user/onboarding` を呼び、成功後 `/` へリダイレクト
- バリデーションエラー表示（`validateUsername` の結果）

### 3-3. ホーム画面 `/`

**責務：** アクティブセッション操作・情報表示・コレクション閲覧

詳細は「4. ホーム画面の構成」を参照。

## 4. ホーム画面の構成

### 4-1. タブ構造

ホーム画面はトップレベルで2つのタブを持つ。

```
/ （ホーム）
├── [ホーム] タブ        ← デフォルト表示
│    └── セッション操作・統計・履歴・ランキング
└── [コレクション] タブ
     └── ユーザーが持つテーマ別タブ
          ├── [鉱夫]    ← miner テーマのフィールドを持つ場合のみ表示
          ├── [漁師]    ← fisher テーマのフィールドを持つ場合のみ表示
          └── [収集家]  ← collector テーマのフィールドを持つ場合のみ表示
```

コレクションのサブタブは、**ユーザーが作成済みのフィールドが持つテーマ**のみ表示する。
持っていないテーマのタブは出さない。将来テーマが増えても同じルールで自動対応する。

### 4-2. 状態

```typescript
// ホーム画面が保持する状態（useState）
type HomePageState = {
  // サーバーから取得したデータ
  homeData: HomeDataResponse | null;
  collectionData: CollectionResponse | null; // コレクションタブ初回表示時に取得

  // アクティブセッション（メモリのみ、DBには終了時だけ保存）
  activeSession: ActiveWorkSession | null;

  // セッション終了後の結果（同一ページ内に表示）
  sessionResult: SaveEndedWorkSessionResponse | null;

  // UI
  activeTab: "home" | "collection";
  activeCollectionTheme: string | null; // 表示中テーマの key
  isLoading: boolean;
};
```

### 4-3. ホームタブのセクション構成

```
ホームタブ
├── ヘッダー
│   ├── ユーザー名
│   ├── レベル表示（Lv xx）+ プログレスバー
│   └── ログアウトボタン
│
├── セッション結果エリア（sessionResult がある場合のみ表示）
│   ├── 有効時間
│   ├── スコア
│   ├── 獲得XP
│   ├── レベルアップ表示（leveledUp === true の場合のみ）
│   ├── 獲得アイテム（droppedItem がある場合のみ）
│   └── [閉じる / 新しいセッションを始める] ボタン
│
├── セッションエリア
│   ├── [idle 時] フィールド選択プルダウン + 開始ボタン
│   ├── [working 時] タイマー表示 + 休憩ボタン + 終了ボタン
│   └── [paused 時] タイマー表示（一時停止中）+ 再開ボタン + 終了ボタン
│
├── フィールド累計一覧
│   └── フィールド名 / テーマ / 累計有効時間 / 累計セッション数
│
├── 最近の履歴
│   └── 最近5件の WorkSession（日時・有効時間・スコア・XP）
│
└── ランキングセクション（折りたたみ可）
    └── 近傍ランキング表示
```

### 4-4. セッション結果の表示ルール

- モーダルや別ページへの遷移は**使わない**
- 終了ボタン押下 → API レスポンス受信 → `sessionResult` に格納 → 結果エリアが現れる
- 結果エリアはセッションエリアの**上に差し込む**（セッション操作エリアを置き換える）
- 「新しいセッションを始める」ボタンで `sessionResult = null` に戻し、idle 状態に戻る
- ページリロードで自然に消える（永続化しない）

### 4-5. コレクションタブのセクション構成

```
コレクションタブ
└── テーマサブタブ（ユーザーの保有テーマのみ）
    └── [選択テーマ] のアイテム一覧
         ├── レアリティ別フィルター（全て / common / rare / epic）
         ├── 取得済み件数 / 全件数
         └── アイテムグリッド
              ├── 取得済み：アイテム名・レアリティ・説明文
              └── 未取得：「???」シルエット表示
```

コレクションタブのデータ（`GET /api/collection`）はタブ初回表示時に取得し、
以降はキャッシュを使う。セッション終了でアイテムが増えた場合は再取得する。

### 4-6. タイマー表示の注意点

- タイマーはフロントのカウントアップ表示であり、真実の状態ではない
- `setInterval` で1秒ごとに `calculateCurrentEffectiveSeconds(activeSession, new Date())` を呼ぶ
- `activeSession` の `startedAt / pausedAt / pauseAccumulatedSeconds` が真実
- ページリロード時はメモリ上のセッションが消えるため、セッションを継続する設計は持たない（MVP）

## 5. コンポーネント分割方針

コンポーネント分割は以下の基準に従う：

- **再利用される** → コンポーネントに切り出す
- **1画面にしか出ない複雑なブロック** → 画面ファイル内に留める
- **ロジックを含む** → ロジックを `lib/` に切り出し、コンポーネントは呼び出しのみ

### 推奨するコンポーネント構成

```
src/
  app/
    components/
      LevelBadge.tsx          # レベル数値 + プログレスバー
      TimerDisplay.tsx        # タイマー数値表示（mm:ss）
      SessionResultCard.tsx   # セッション結果表示エリア
      FieldCard.tsx           # フィールド累計一覧の1件
      RankingList.tsx         # 近傍ランキング表示
      CollectionGrid.tsx      # アイテムグリッド（テーマ1つ分）
      CollectionItemCard.tsx  # アイテム1件（取得済み/未取得の分岐）
```

### コンポーネントの入出力原則

- Props のみで描画できるようにする（useState を持たない）
- コールバック関数は `onXxx` の形で受け取る
- API 呼び出しは親画面コンポーネント（`app/page.tsx`）で行い、結果を Props として渡す

## 6. データフロー

```
サーバー（API）
  ↓ fetch
app/page.tsx (useState + useEffect)
  ↓ props
表示コンポーネント (stateless)
```

- クライアント側の計算（タイマー、スコアプレビュー）は `src/lib/` の関数を呼ぶ
- サーバー側の計算（確定値）は API レスポンスをそのまま使う
- 状態を2箇所で持たない（フロントのカウントは表示用、DB の値が確定値）

## 7. スタイリング方針

- 現状は `globals.css` のみで最小限のスタイル
- MVP では見た目の作り込みより構造の正しさを優先する
- コンポーネントにスタイルをインラインで書かない
