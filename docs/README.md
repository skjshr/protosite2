# ドキュメント構成

## Codex・Claude が実装前に読む順序

1. `project/coding-standards.md` — 命名・責務分離・コメント方針（必須）
2. `project/architecture.md` — 技術スタック・レイヤー構成（必須）
3. `requirements/product-spec.md` — プロダクト思想・MVP範囲（必須）
4. 実装対象の `features/` ファイル — 体験レベルの仕様
5. 実装対象の `design/` ファイル — 実装詳細
6. `tasks/` の作業指示ファイル — 具体的な実装タスク

---

## フォルダ構成

```
docs/
  README.md              このファイル（案内・読む順序）
  project/               プロジェクト共通ルール（全タスクで常に読む）
  requirements/          要件定義・プロダクト思想・MVP範囲
  features/              機能設計（何を・なぜ。体験・スコープレベル）
  design/                詳細設計（どう実装するか。DB・API・アルゴリズム）
  test/                  テスト設計（方針・ケース）
  tasks/                 Codex作業指示（タスクごとのmd）
  decisions/             設計判断の記録（なぜその設計にしたか）
```

---

## 各フォルダの役割

### project/
- 命名規則・責務分離・コメント方針などの共通規約
- 技術スタック・ディレクトリ構造・レイヤー境界
- **全タスクで読む**

### requirements/
- プロダクトの思想・スコア設計の哲学
- MVP の範囲とやらないこと
- **初回と判断に迷ったときに読む**

### features/
- 各機能の体験設計（ユーザー視点）
- 画面遷移・状態変化の概要
- 実装詳細は含まない（design/ を参照）
- **対象機能の実装前に読む**

### design/
- 実装レベルの詳細設計
- DB スキーマ・API 仕様・アルゴリズム・型定義
- 責務分離・ファイル配置の指針
- **実装するファイルに対応する設計書を読む**

### test/
- テスト方針・優先度
- 機能別テストケース
- **テスト実装時に読む**

### tasks/
- Codex への具体的な作業指示
- 1ファイル1タスク
- 完了したタスクは `_done/` サブフォルダへ移動する

### decisions/
- 設計判断の記録（Architecture Decision Record 風）
- 「なぜこの設計を選んだか」「何を捨てたか」
- 後から見直したときの根拠として使う
